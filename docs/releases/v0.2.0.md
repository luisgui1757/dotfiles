# v0.2.0 — Nix package ownership and transactional v0.1 migration

> Release candidate notes. Publish these only with the official annotated
> `v0.2.0` tag after the release evidence gate in `docs/UPGRADING.md` passes.

## Breaking contract changes

- Apple Silicon is the sole macOS architecture in the v0.2.0 product contract;
  no other Darwin configuration, installer path, or migration path is shipped.
- Nix owns the required POSIX package layer: nix-darwin on macOS and
  package-only Home Manager on Linux/WSL. Chezmoi still owns every config
  target. Native Windows remains non-Nix.
- WSL is explicitly split-host: Windows setup owns host Terminal/font/clipboard
  pieces and the distro owns its Linux Nix/config state.
- Windows resolves UserProfile, LocalApplicationData, Documents, and the active
  PowerShell profile independently, including redirected and alternate-drive
  known folders. Packaged and portable Windows Terminal settings are separate
  transactions.
- Required CI identities move from runner-image names to stable logical names
  through a post-merge owner transaction; both name sets are emitted during the
  transition.

## Upgrade requirement

Do not update a v0.1.0 checkout in place. Its POSIX targets are live symlinks
into that checkout, so changing Git first crosses the config mutation boundary
before backup.

Use the side-by-side, exact-tag migration in
[`docs/UPGRADING.md`](../UPGRADING.md). It verifies both annotated releases and
the live historical config, captures private recovery state, applies only the
reversible Nix/config-file core, defers chezmoi run scripts and native tools,
publishes and rolls back only from digest-bound exact-commit trees, and retains
v0.1.0 until explicit acceptance.
Failure or interruption after package activation restores the prior config and
removes the first nix-darwin/Home Manager activation. Windows restores exact
pre-migration Terminal bytes and v0.1.0 config from its validated frozen release
tree, so rollback does not read mutable checkout bytes.

## Release identity

The checked-in migration tools require the tag name `v0.2.0` and compare its
local annotated tag object and peeled commit to the official remote at runtime;
there is no pre-release commit placeholder to trust. They reject a branch,
lightweight tag, dirty checkout, unofficial remote, or moved tag. The observed
tag object and peeled commit are recorded in the append-only review ledger only
after GitHub publishes them.

## Evidence required before publication

- full local and hosted required checks on the exact release candidate;
- exact v0.1.0 fixture (`015617362830280bf85c7142e69d0681d376d453`)
  through apply, injected later failure, rollback, retry, and acceptance;
- Apple Silicon owner-host, native Linux, real WSL2 split-host, conventional and
  redirected Windows, and divergent packaged/portable Terminal upgrade rows;
- public secret scan across `v0.1.0..v0.2.0` and the generated proof artifacts;
- after merge only, cache-free stable-check proof and owner safeguard cutover.

Until those rows are complete and the annotated tag exists, v0.1.0 remains the
only user-facing release.
